Becoming a Better Programmer

Sep 9th, 2012

This is something I’ve asked myself, as well as being asked this.

Whether you’re programming for a living or as a hobby, you’ll want to consider how to ‘hack’ yourself to improve your efficiency and effectiveness. Over the past couple of years, I’ve interviewed quite a lot of people, and sadly, I think many of them were not invested in themselves and their learning habits and aren’t able to pick things up as quickly as todays environment demands.

And I don’t consider myself a great programmer. I’ve improved markedly over the past few years, but I’ve still got more to learn and improve. I’ve been leveling up, but I haven’t topped out yet :)

Here follows a list of some of the things I think are important in no particular order (I love lists). Some of this is Ruby specific, but some is generally applicable.

Read API documentation.

For example, reading the Ruby library documentation for Enumerable is absolutely critical as is reading and understanding the Ruby Guides and ActiveRecord API docs. You cannot get by reading just blogs and watching screencasts.

When there is an error, read and understand what the ruby interpreter is trying telling you.

My initial work in ruby was quite frustrating, with my only previous experience in standard languages such as C, C++ and PHP. The ruby language is grammatically complex and so the interpreter isn’t able point to this line and say, “this is your problem, right here!” as the compilers I was used to.

Instead I got multi-page long stacktraces (this was in Rails 2) that were bewildering. As with many things, embrace the pain, and your brain will start to filter out the irrelevant information and start to pick out the important bits.

When you run into a problem, step back and try to understand why it’s happening.

The two most important questions I try to answer when encountering unexpected behavior (aka bug or defect) are WHAT and WHY.

What is happening?

This requires understanding the interactions and the context that lead up to the error. This often why when I get a bug report, if it is not detailed enough, have to either throw it back with a ‘need more info’ or spend hours (sometimes days) trying to reproduce the problem. At work since we don’t have access to production logs (see ops anti-patterns ), this takes much longer than it should.

Why is it happening?

This needs to be understood in order to take correct action to resolve the issue, and is often the harder question to answer. Especially if you’re in code that you haven’t written yourself, whether that is 3rd party library code, or code inherited in a large application.

Know your tools

This is something that the pragmatic programmer book mentions. In my opinion, you need to understand at least one editor well enough to not waste time with all the text processing you’ll need to do.

IMO, the most important thing you’ll need to be able to do is quickly and efficiently search for strings in files, across files, as well as string patterns using regexes.

If you can learn more than one editor well, even better. I use both vim and Sublime Text2 for everyday text editing tasks. Previously, I used the Netbeans IDE because it actually parsed the script files into an AST which allowed me to follow symbols back to definitions, even jumping into gems. I learned quite a bit about ActiveRecord and how rails worked by doing this.

But I seldom do that any longer. Finding ruby code is as easy as global find in Sublime, and Ack in vim. If I need to follow code into a gem, bundle open is da shit.

Debugging techniques

This can be an entire post by itself, so I’ll just touch on a couple of things to check up on.

Debuggers

  • In ruby, require 'debugger', not the outdated ‘ruby-debug’. I often use the debugger as a first step to determine why something is not behaving as expected. Just drop in a debugger line trigger the debugger. Just be aware the debugger stops execution at the line following the debugger call. So if you put the debugger line at the end of a block, you’ll end up in an entirely different function than you expected.
  • In JS, just put debugger anywhere and in Firefox or Chrome it will just stop on that line.

Logging

  • Ruby: you can log use Rails.logger.error , use puts, or even just raise an exception with the variable you’re interested in.
  • JS: console.log – pretty simple, but don’t forget to take it out, because it will cause a js exception in IE. Another tip: pass in js objects that you want to inspect as separate parameters, not coercing them into strings. e.g. console.log("some value = ", some_value) instead of console.log("some value" + some_value)

Have the right attitude

It can be hard at times, especially when starting out your career to absorb and make use of all the information available.

While I’m not an advocate of being uncontrollably positive, I don’t think having a negative attitude directed at yourself or at a problem is constructive. Try to take each challenge as it comes, knowing that you will probably solve it sooner or later. Take in the journey, knowing that the struggle will improve your skills later.

Related to having the right attitude, in a commercial environment you want to avoid being personally invested in code you have written (ie. no ego), but take responsibility for all code that you come across, regardless of if came from you or not.

Put another way, avoid immediately looking to refactor throw out other peoples code that you may not understand or agree with. There may be thing or two to learn, and it may have been written that way for a good reason that is not immediately obvious. Not to say all code is good and never needs to be refactored.

Searching properly

Improving you google-fu will help you avoid wasting time. Unfortunately, I can’t explain how I search for things, but if what you’re looking for doesn’t come up in the first page of results, your search terms need adjusting. Looking through page after page of search results probably won’t be a good use of your time.

Read source code

I think this is probably the biggest reason I’ve improved my coding and design skills. Read lots of code. Lots of other people’s code. Understand it. Modify it to understand it.

Doing this well will reduce the amount you will need to google for an answer. These days if I’m using 3rd party code, instead of checking google or stackoverflow, I’ll just bundle open the gem to see how it works and answer the question that way.
